# Security Center

## Purpose

Security Center is the single screen where platform security team members review active alerts, investigate suspicious activity, manage platform admin sessions, and take security response actions. It surfaces alerts that were automatically generated by the platform's detection pathways and provides the acknowledge/resolve workflow for those alerts.

## Alert Detection (summary)

Alerts are not created manually. They are generated by three automated backend pathways:

1. **MediatR domain event handlers** â€” module events (login failures, payment failures, identity verification spikes) trigger `AlertCreationHandler` when thresholds are crossed
2. **Background health check jobs** â€” `PlatformHealthCheckJob` polls service endpoints every 2 minutes; `ResourceUtilizationCheckJob` checks CPU/memory/storage every 5 minutes
3. **Webhook handlers** â€” Stripe/Paddle/PayHere payment failure events create billing alerts

See [[developer-platform/modules/dashboard/end-to-end-logic|Dashboard End-to-End Logic]] for the complete alert condition catalog and severity classification rules.

## Capabilities

### Alert Management
- View all active (unresolved) alerts across all tenants and platform systems, filtered by severity, status, tenant, module, and date range
- View alert detail: full condition description, affected tenant, source module, related audit events
- **Acknowledge** â€” mark that an operator has seen the alert and is investigating (does not resolve)
- **Resolve** â€” close the alert with a resolution type and note (Critical severity requires a minimum 20-character written note)
- View resolved and auto-resolved alert history

### Session Management
- View all active platform admin sessions with IP address, login time, and expiry
- **Revoke** a specific session â€” takes effect immediately; revoking own session logs out the current operator
- Revoke all sessions for a specific account (also accessible from Platform Access)

### Suspicious Activity
- View below-threshold security events from the audit log that did not reach alert creation (e.g., 5 failed logins â€” suspicious but not yet brute-force threshold of 10)
- Link directly to Audit Console for full event detail

## Key Rules

- Every read of the Security Center is itself audit-logged â€” reading sensitive data is a security event
- Token hashes stored in `dev_platform_sessions` are never returned by any API response
- Alert resolution notes are plaintext â€” operators must not include credentials or secrets in resolution notes
- Alerts cannot be suppressed or deleted â€” only acknowledged and resolved. The full creation-to-resolution trail is permanent.
- Session revocation of own session logs out the current operator immediately
